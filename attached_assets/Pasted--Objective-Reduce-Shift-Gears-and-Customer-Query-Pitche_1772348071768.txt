# Objective
Reduce Shift Gears and Customer Query Pitches response times from ~15-30 seconds to under 7-8 seconds by addressing the actual root causes found in production deployment logs.

# How It Works Today (End-to-End Explanation)

When you're on a live call, here's what happens every time the transcript changes:

### Frontend (client/src/components/shift-gears.tsx)
1. **Wait:** A 2-second throttle + 0.5-second debounce fires before ANY request is sent (~2.5s minimum perceived delay)
2. **Shift Gears request:** POST to `/api/conversations/:sessionId/shift-gears`
3. **Query Pitches request:** POST to `/api/conversations/:sessionId/query-pitches`
4. **Meanwhile:** Every transcript utterance also POSTs to `/api/conversations/:sessionId/messages` (the main "sales response" AI call)

### Backend - What Each Request Does
**For Shift Gears & Query Pitches:**
1. **Semantic search** (~200-500ms): Calls OpenAI embedding API to find relevant training documents
2. **Build system prompt** (~50ms): Combines training context (capped at 3,000 chars) + coaching instructions → ~3,158 prompt tokens total
3. **AI model call** (THE BOTTLENECK): Sends to DeepSeek → **15-20 seconds** per call

**For /messages (generateSalesResponse):**
1. **Sales framework context** (~200ms): Builds conversation memory and framework
2. **Training context** with **30,000 token limit** (~500ms): Fetches ALL training documents
3. **AI model call**: Sends ~9,400 prompt tokens to DeepSeek → another 10+ seconds
4. Has a 10-second timeout (often hits it → returns fallback)

### The Core Problem: ALL of these calls hit the same DeepSeek API simultaneously
During a single conversation turn, the system fires 3-5 concurrent AI calls to the same DeepSeek API endpoint. DeepSeek processes them sequentially per API key, so they **queue up** and each one waits for the previous to finish.

# Root Causes (From Production Logs)

## Root Cause 1: DeepSeek is slow — and there is NO fast model fallback for it
**Evidence:** `⚡ ShiftGears AI call: 15154ms | Model: deepseek-chat`

The fast-model switching code does: if model includes 'gpt-4' → use gpt-4o-mini, if 'claude' → haiku, if 'gemini' → flash. But DeepSeek model is `deepseek-chat` — it matches NONE of these, so it stays on `deepseek-chat`. No speed improvement at all.

## Root Cause 2: generateSalesResponse uses 9,400 prompt tokens AND is labeled as shift_gears
**Evidence:** `Recorded 9486 tokens (9381 prompt + 105 completion) for deepseek - shift_gears`

This separate function (`/messages` endpoint) fetches training context with a **30,000 token limit** (vs 3,000 for Shift Gears), producing ~9,400 prompt tokens. It records usage under `shift_gears` label, making it look like Shift Gears is consuming all those tokens. This call competes for the same DeepSeek API.

## Root Cause 3: Multiple concurrent AI calls saturate the API
**Evidence:** During a single conversation turn, logs show 3-5 overlapping requests:
- generateSalesResponse (9,400 tokens)
- generateShiftGearsTips (3,158 tokens)
- generateQueryPitches (~3,000 tokens)
- one-liners (takes 15 seconds!)
All going to the same DeepSeek endpoint simultaneously.

## Root Cause 4: JSON parse errors causing wasted computation
**Evidence:** `JSON parse error: SyntaxError: Unterminated string in JSON at position 2970` (multiple occurrences)

DeepSeek frequently returns truncated/malformed JSON at max_tokens=600, requiring retry or fallback — wasting time.

# Tasks

### T001: Reduce generateSalesResponse training context from 30,000 to 3,000 tokens
- **Blocked By**: []
- **Details**:
  - `generateSalesResponse` fetches 30,000 tokens of training context (line 1327), producing 9,400+ prompt tokens per call
  - Reduce to 3,000 tokens (matching Shift Gears) since this is a real-time coaching response, not a document
  - This immediately cuts the DeepSeek processing time by ~60% for this function
  - Files: `server/services/openai.ts` (lines 1327, 1342)
  - Acceptance: Token count for generateSalesResponse drops from ~9,400 to ~3,000 prompt tokens

### T002: Add server-side concurrency control — skip duplicate in-flight requests
- **Blocked By**: []
- **Details**:
  - Add an in-flight request tracker per session per feature (shift_gears, query_pitches)
  - If a Shift Gears request is already processing for a session, immediately return the previous cached result instead of queueing another AI call
  - This prevents 3-5 concurrent DeepSeek calls from stacking up
  - Files: `server/services/openai.ts` (generateShiftGearsTips, generateQueryPitches)
  - Acceptance: Only 1 AI call per feature per session at a time; concurrent requests get the cached result

### T003: Add 8-second hard timeout for Shift Gears and Query Pitches AI calls
- **Blocked By**: []
- **Details**:
  - Currently there's no timeout on the AI call itself within generateShiftGearsTips/generateQueryPitches — the user waits 15-30 seconds
  - Add a Promise.race with an 8-second timeout: if DeepSeek doesn't respond in 8s, return the last cached tips or a graceful fallback
  - This ensures the user NEVER waits more than ~8 seconds
  - Files: `server/services/openai.ts` (around lines 3426-3441 for ShiftGears, 3629-3644 for QueryPitches)
  - Acceptance: Response time capped at 8 seconds maximum

### T004: Cache last-good response per session and serve during timeouts/pending calls
- **Blocked By**: [T002, T003]
- **Details**:
  - Maintain a per-session cache of the last successful Shift Gears tips and Query Pitches results
  - When a timeout occurs (T003), return the cached response
  - When a request is blocked by an in-flight one (T002), return the cached response
  - This ensures the UI always shows SOMETHING useful while waiting
  - Files: `server/services/openai.ts`
  - Acceptance: No more empty/loading states lasting 15-30 seconds

### T005: Increase max_tokens from 600 to 700 to prevent JSON truncation errors
- **Blocked By**: []
- **Details**:
  - Production logs show frequent JSON parse errors: `Unterminated string in JSON at position 2970`
  - DeepSeek is cutting off JSON at exactly 600 tokens, producing invalid JSON
  - Increase to 700 so the JSON can complete properly, avoiding wasted computation on parse failures
  - Files: `server/services/openai.ts` (line 3434)
  - Acceptance: Fewer JSON parse errors in production logs

### T006: Reduce temperature for faster AI sampling
- **Blocked By**: []
- **Details**:
  - Current temperature is 0.12 for ShiftGears and 0.15 for QueryPitches
  - Lower to 0.05 for both — for real-time coaching, deterministic responses are fine and faster to generate
  - Files: `server/services/openai.ts` (lines 3433, 3636)
  - Acceptance: Marginally faster AI responses
